# Phase 4 — Distributed Test Execution via Local Agents

**Status:** Planned, not implemented. Pick up in a fresh Claude Code session.

---

## Context

After Phases 1-3, the platform has:
- Centralized SQLite storage
- Multi-user dashboard (React + WebApi)
- Distributed recording (Runner CLI posts recorded tests to the shared server)
- Docker / self-contained server deployment

**The unsolved problem:** the server cannot execute web UI or desktop UI tests.
- **Desktop tests** need an interactive Windows desktop + the target WinForms app installed.
- **Web UI tests** need Chromium, which requires Windows Media Foundation — not installable in Windows Server Core containers (DISM fails with no package source; manual DLL copy fails signature verification).

Today QA engineers have to run the Runner CLI manually via command line to execute these tests. We want them to use the dashboard like they do for API/aseXML tests.

**The answer:** turn the Runner CLI into a long-running **agent** on each QA engineer's local machine. The agent polls the server for queued jobs, claims them, executes locally (where the browser/desktop app is available), and posts results back. The dashboard triggers runs exactly as it does today; the server either executes locally (API/aseXML) or enqueues for an agent (web/desktop).

Single UI for QA engineers, test execution happens on the right machine every time, zero CLI knowledge required for day-to-day usage.

## Data Model

Two new SQLite tables.

### `agents`
Long-lived registration of each local Runner instance.

```sql
CREATE TABLE agents (
    id            TEXT PRIMARY KEY,     -- GUID, generated by agent on first start
    name          TEXT NOT NULL,        -- "Alice-PC", user-supplied
    user_id       TEXT,                 -- FK to users.id (owner of the agent's API key)
    capabilities  TEXT NOT NULL,        -- JSON array: ["UI_Web_Blazor","UI_Web_MVC","UI_Desktop_WinForms"]
    version       TEXT,                 -- Runner version for diagnostics
    status        TEXT NOT NULL,        -- Online | Offline | Busy
    last_seen_at  TEXT NOT NULL,        -- ISO8601, updated on heartbeat
    registered_at TEXT NOT NULL
);
```

### `run_queue`
Jobs waiting to be picked up, or claimed/completed.

```sql
CREATE TABLE run_queue (
    id             TEXT PRIMARY KEY,           -- matches the runId
    module_id      TEXT NOT NULL,
    test_set_id    TEXT NOT NULL,
    objective_id   TEXT,                       -- null = run whole test set
    target_type    TEXT NOT NULL,              -- "UI_Web_Blazor" | ... (used to match agent capabilities)
    mode           TEXT NOT NULL,              -- "Reuse" | "Rebaseline" | "VerifyOnly"
    requested_by   TEXT,                       -- user.id who triggered it
    status         TEXT NOT NULL,              -- Queued | Claimed | Running | Completed | Failed | Cancelled
    claimed_by     TEXT,                       -- agent.id, null when Queued
    claimed_at     TEXT,
    completed_at   TEXT,
    error          TEXT,
    request_json   TEXT NOT NULL,              -- Full RunRequest serialised so the agent has everything
    created_at     TEXT NOT NULL
);

CREATE INDEX idx_run_queue_status_target ON run_queue (status, target_type);
```

**Status transitions:** `Queued → Claimed → Running → (Completed | Failed)`. Cancelled is a terminal state for jobs abandoned before claim (e.g. user cancelled from UI).

## API Endpoints

All require `X-Api-Key` (reuse existing auth middleware). Agent endpoints require the key of the user who owns the agent.

### Agent lifecycle

| Method | Route | Purpose |
|---|---|---|
| `POST` | `/api/agents/register` | Register a new agent or re-register on startup. Body: `{name, capabilities[], version}`. Returns `{agentId}`. |
| `POST` | `/api/agents/{id}/heartbeat` | Keep-alive. Updates `last_seen_at` and `status`. Body: `{status}`. Server marks agent Offline if no heartbeat in 2 min. |
| `DELETE` | `/api/agents/{id}` | Graceful deregister (Ctrl+C on Runner). |
| `GET` | `/api/agents` | List all agents (for dashboard). Returns name, status, capabilities, last seen, current job. |

### Queue operations (agent-side)

| Method | Route | Purpose |
|---|---|---|
| `GET` | `/api/queue/next?capabilities=UI_Web_Blazor,UI_Web_MVC&agentId={id}` | Atomic claim-oldest-matching. Returns `{jobId, request}` or `204 No Content` if no match. Uses `UPDATE ... WHERE status='Queued' ... RETURNING *` (SQLite 3.35+) or transaction. |
| `POST` | `/api/queue/{jobId}/progress` | Agent reports in-progress status. Body: `{status: "Running"}`. Used to flip `Claimed → Running`. |
| `POST` | `/api/queue/{jobId}/result` | Agent reports terminal result. Body: `{success, error, executionRunId}`. Server marks the job Completed/Failed. |

### Queue operations (dashboard-side)

| Method | Route | Purpose |
|---|---|---|
| `GET` | `/api/queue` | List all queue entries (Queued, Claimed, Running, recent Completed). For the dashboard's queue panel. |
| `DELETE` | `/api/queue/{jobId}` | Cancel a Queued job. Fails if already Claimed. |

### Modified: existing run trigger endpoints

`POST /api/runs` and `POST /api/modules/{id}/run` — add branching logic:

```
if (targetType is UI_Web_Blazor|UI_Web_MVC|UI_Desktop_WinForms) {
    insert into run_queue, return 202 { runId, status: "Queued" }
} else {
    existing behaviour — execute in-process
}
```

Dashboard polls `/api/runs/{runId}/status` — that endpoint already exists; extend it to check `run_queue` first, then fall back to `RunTracker`.

## Runner CLI Agent Mode

New flag: `--agent`. Long-running mode that turns the Runner into a worker.

### Command

```powershell
dotnet run --project src/AiTestCrew.Runner -- --agent --name "Alice-PC" [--capabilities UI_Web_Blazor,UI_Web_MVC,UI_Desktop_WinForms]
```

`--capabilities` is optional — defaults to "all the agents this Runner has DI-registered" (typically all three UI targets).

`ServerUrl` and `ApiKey` must be set in `appsettings.json` (existing Phase 3 config).

### Behaviour

1. **Startup:**
   - Load/generate a persistent `agentId` (stored in local `.agent-id` file so the same machine keeps the same ID across restarts)
   - Call `POST /api/agents/register` with name, capabilities, version
   - Print agent ID and "connected to {serverUrl}"

2. **Main loop (until Ctrl+C):**
   - Every 30s: send heartbeat with current status
   - Every 10s (when idle): `GET /api/queue/next?capabilities=...`
   - If a job is returned:
     - Set status to Busy, heartbeat immediately
     - Reconstruct a `RunRequest` from `request_json`
     - Call the existing orchestrator path (`TestOrchestrator.RunAsync`) synchronously
     - On completion, call `POST /api/queue/{jobId}/result`
     - Set status back to Online

3. **Graceful shutdown:**
   - Ctrl+C → finish current job if running, then `DELETE /api/agents/{id}`
   - If killed ungracefully, the server detects via heartbeat timeout (2 min) and marks Offline

### New files

| File | Purpose |
|---|---|
| `src/AiTestCrew.Runner/AgentMode/AgentRunner.cs` | Main agent loop — register, heartbeat, poll, execute |
| `src/AiTestCrew.Runner/AgentMode/AgentClient.cs` | HTTP wrapper around the queue/agent endpoints (similar to RemoteHttpClient) |
| `src/AiTestCrew.Runner/AgentMode/JobExecutor.cs` | Bridges a dequeued job to the existing orchestrator — reconstructs RunRequest, calls RunAsync, captures result |

### Modified files

- `src/AiTestCrew.Runner/Program.cs` — add `--agent`, `--name`, `--capabilities` flags. Short-circuit into AgentRunner instead of the existing CLI flow. Build the host (DI, orchestrator, agents) as normal but don't exit after one run.

## Design Decisions (record for the implementer)

1. **Polling vs SignalR.** Polling (10s for jobs, 30s for heartbeat) is simpler and "good enough" for a team of 5-15. SignalR would add a persistent connection, require WebSocket support everywhere, and make the agent's connection lifecycle more complex. Start with polling; revisit if latency feels bad.

2. **Atomic claim.** Use `UPDATE run_queue SET status='Claimed', claimed_by=? WHERE id=(SELECT id FROM run_queue WHERE status='Queued' AND target_type IN (...) ORDER BY created_at LIMIT 1) RETURNING *`. SQLite 3.35+ supports `RETURNING`. If the Microsoft.Data.Sqlite bundled version is older, use a transaction with `BEGIN IMMEDIATE`.

3. **Agent capabilities.** Declared by the Runner on startup. In practice, all three UI agents (`UI_Web_Blazor`, `UI_Web_MVC`, `UI_Desktop_WinForms`) are always registered together because they all need Windows. But keeping the capability set explicit makes the design extensible (future: Linux agents for Playwright-only).

4. **Heartbeat timeout.** 2 min default. Balance between fast failover and tolerating brief network blips. Configurable via `AgentHeartbeatTimeoutSeconds` in server config.

5. **What if no agent is online?** The job stays Queued. When an agent with matching capabilities comes online, the first `/api/queue/next` call picks it up. Dashboard shows a clear "no agent available" message so users don't wait forever.

6. **Job cancellation during execution.** Out of scope for Phase 4. Cancel only supported before claim. If needed, add a `cancelled` flag the agent checks periodically and add CancellationToken plumbing through the orchestrator.

7. **Agent owner vs run requester.** The `claimed_by` column is the agent; the `requested_by` column on `run_queue` is the user who clicked Run. Dashboard shows both: "Bob ran it on Alice-PC".

8. **Offline server resilience.** If the agent loses connection to the server, it should log, keep trying to reconnect, not crash. The server doesn't need to know about the agent during downtime — agents are stateless from the server's POV, they just re-register on reconnection.

9. **Security.** Agent registration is authenticated via the owning user's API key. `claimed_by` should match the `agentId` the caller supplies — prevents one agent from stealing another's claim.

## Verification Plan

End-to-end test after Step 5:

1. **Server running in Docker.** Dashboard accessible at http://localhost:5050.
2. **Agent running on host:** `dotnet run --project src/AiTestCrew.Runner -- --agent --name "MyPC"` in a second terminal. Agent logs "Registered as {guid}, polling for jobs...".
3. **Dashboard check:** Agents panel shows "MyPC - Online - Blazor, MVC, Desktop".
4. **Trigger a Blazor test** via the dashboard Run button. Status should flip: Queued → Running on MyPC → Completed. Results appear in run history.
5. **Trigger a desktop test.** Same flow — picked up by the same agent.
6. **Trigger an API test.** Executes in the Docker container as before (not queued), results appear immediately.
7. **Stop the agent** (Ctrl+C). Agents panel shows it Offline within 2 min.
8. **Trigger another Blazor test** while no agent is online. Dashboard shows "Queued — waiting for agent with UI_Web_Blazor capability".
9. **Restart the agent.** Queued job picks up automatically within 10s.

If all nine pass, Phase 4 is complete.

- Scheduled runs (Phase 4+ enhancement from the original plan)
- SignalR / push notifications (polling is sufficient at this scale)
- Agent groups / routing rules (e.g. "staging tests → staging agents") — would layer on top of capabilities if needed later
